# SARA Industrial Deployment Guide

SARA is designed for safe, automated deployments. This guide ensures your industrial site is optimized for performance and secured against common web vulnerabilities.

## 1. Automated CI/CD (GitHub Actions)
Create `.github/workflows/deploy.yml` in your project root. SARA's dependency graph makes caching extremely effective.

```yaml
name: Deploy SARA Site
on: [push]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: haskell/actions/setup@v2
        with:
          ghc-version: '9.2.8'
      
      - name: Cache Build
        uses: actions/cache@v3
        with:
          path: |
            dist-newstyle
            _build
          key: ${{ runner.os }}-sara-${{ hashFiles('**/*.hs') }}

      - name: Build
        run: cabal run sara -- build
      
      - name: Upload Artifact
        uses: actions/upload-pages-artifact@v1
        with:
          path: _site
```

## 2. Server Configuration (Nginx)
When deploying the static output, ensure you serve with correct MIME types for modern formats like **AVIF** and **WebP** generated by SARA.

```nginx
server {
    listen 80;
    root /var/www/sara-site;

    location / {
        index index.html;
        # Enable View Transitions and smooth navigation
        add_header View-Transition "same-origin";
    }

    # SARA's search index is immutable per build
    location ~* \.json$ {
        expires 1h;
        add_header Cache-Control "public";
    }
}
```

## 3. Security Hardening
*   **The Local Guard**: SARA's health dashboard (`/_sara`) is automatically blocked for non-loopback IPs. Do not attempt to bypass this in production.
*   **Permissions**: Ensure the SARA binary does not run as `root` in your build environment. `ShellGuard` will automatically block shell injection attempts, but standard Linux hygiene is still required.

## 4. Inverted Index Performance
For sites over 10,000 pages, ensure your build server has at least 2GB of RAM. SARA's incremental indexing uses partial JSON fragments which can be memory-intensive during the final merge phase.
