SARA Static Site Generator — Severe Code Audit
Repo: Bombay-Boyz/sara · Language: Haskell (GHC2021) · Build: Shake · Templates: Mustache (stache)

1. Critical: unsafePerformIO Misuse
SARA.Routing.Engine (lines ~55–65):
haskellcompiled = unsafePerformIO $ RE.compile RE.compBlank RE.execBlank pat
...
let matches = unsafePerformIO $ RE.execute compiledRegex pathText
SARA.Security.RegexGuard:
haskellcase unsafePerformIO $ compile compBlank execBlank t of
SARA.LiveReload.Server:
haskellclientCounter :: IORef Int
clientCounter = unsafePerformIO $ newIORef 0
This is the worst issue in the codebase. unsafePerformIO is justified exactly once in a Haskell codebase: top-level IORef/MVar global state with {-# NOINLINE #-}. The server counter at least has {-# NOINLINE #-}. The routing/regex usages are completely unjustified — resolveRoute must live in IO or Action. The regex compilation result is not referentially transparent; it depends on a mutable internal PCRE state. GHC can inline, duplicate, or float these calls — the behavior is undefined. This is not "controlled purity" as the comment claims. It is a lie the type system cannot catch.

2. Critical: error Used as a Build-Failure Mechanism
In SARA.Internal.Planner (genRender):
haskellLeft err -> error $ T.unpack (renderAnyErrorColor (AnySaraError err))
Left err -> error $ T.unpack (renderAnyErrorColor (AnySaraError err))
Left err -> error $ show err
In SARA.DSL (loadData):
haskellNothing -> error $ "Failed to parse JSON: " ++ path
Left err -> error $ "Failed to parse YAML: " ++ path ++ ": " ++ show err
_       -> error $ "Unsupported data format: " ++ ext
In SARA.Security.PathGuard:
haskellguardPathUnsafe root path = case guardPath root path of
  Right p -> p
  Left  e -> error $ "guardPathUnsafe: " ++ ...
The monad stack (SaraM) already has ExceptT [AnySaraError]. There is zero reason to call error inside planRules. These are partial functions that throw impure exceptions and bypass the type system completely. Every single one of these should be throwError. The loadData function drops out of the monad silently for unsupported file extensions. The guardPathUnsafe function is a loaded gun exported from a security module.

3. Critical: The validateSEO Function is a No-Op
haskellvalidateSEO :: Item 'Unvalidated -> SaraM (Item 'Validated)
validateSEO item = return $ Item
  { itemPath = itemPath item
  , itemRoute = itemRoute item
  ...
  }
The phantom type trick with ValidationState is the most architecturally promising thing in the codebase. The entire point is that you can't render an Item 'Unvalidated — you must pass through a validator. But the validator does absolutely nothing. It just unsafely coerces the phantom tag. The type system promise is being made to the user ("your item is validated") while silently being violated. This is worse than not having the type at all — it creates false confidence.

4. Serious: LQIP Token Injection is a Hand-Rolled XSS Vector
In SARA.DSL:
haskelltoken = "__LQIP__:\" <> src <> \"__\"
in "<picture class=\"lqip\" style=\"background-image: url(" <> token <> ")\">"
The src value comes from frontmatter, which is HTML-escaped… but is then inlined into a CSS url() expression inside an inline style attribute. HTML-escaping is insufficient for CSS context. A user with a frontmatter key like:
yamlsrc: "x);expression(alert(1))//"
…will break IE and old browser CSS parsers. Additionally, findLQIPTokens is a hand-rolled parser that fails on edge cases (nested __, paths with special chars, etc.) and processes the token with T.drop 8 after T.breakOn "__LQIP__:" which is off-by-one — "__LQIP__:" is 9 characters, not 8.

5. Serious: escapeHtml Has Wrong Ordering
haskellescapeHtml t = T.replace "<" "&lt;"
             . T.replace ">" "&gt;"
             . T.replace "&" "&amp;"
             ...
Replacements are applied right-to-left in . composition, meaning & is escaped last, so &lt; in the input becomes &amp;lt; — double-escaped — while < in input first becomes &lt; and is then left alone. This is backwards. & must be escaped first. Every Haskell novice makes this mistake. The correct order is & → < → > → " → '.

6. Serious: detectFormat is Fooled by JSON Document Bodies
haskelldetectFormat t
  | "{" `T.isPrefixOf` t = FmJSON
A Markdown file whose body (with no frontmatter) begins with { — e.g., a code example, a Liquid template remnant, or any JSON snippet — will be misdetected as JSON frontmatter and fail to parse. Hugo and Hakyll both require { to appear on a line by itself as { followed by a newline. The splitJSON parser is also dangerously naïve: it looks for }\n anywhere in the file, not the top-level closing brace.

7. Serious: match Returns an Empty List, Silently
haskellmatch :: GlobPattern -> (FilePath -> SaraM (Item 'Validated)) -> SaraM [Item 'Validated]
match g f = do
  tell [RuleMatch g (void . f)]
  return []  -- ← always empty
The user code in site.hs does:
haskellposts <- match (glob "posts/*.md") $ \file -> do ...
buildSearchIndex "search-index.json" posts  -- posts is always []
The comment -- always empty in the implementation reveals this is known. The DSL is a lie: it promises items but delivers nothing. The buildSearchIndex will always generate an empty index. Downstream features (RSS, sitemap, search) all silently fail with zero items if they depend on match's return value. This is the single biggest logic bug in the codebase.

8. Moderate: The Monad Stack is Wrong for Shake Integration
haskellnewtype SaraM a = SaraM
  { unSaraM :: ReaderT SaraEnv (WriterT [RuleDecl] (ExceptT [AnySaraError] IO)) a }
WriterT [RuleDecl] accumulates a list. This means every tell is an O(n) left-associated append — a classic Haskell performance pitfall. For large sites with hundreds of rules, this degrades. Use DList or Seq, or better yet — since all rules must be known at planning time — collect rules into an IORef [RuleDecl] in SaraEnv and eliminate the WriterT layer entirely. The current stack also mixes IO (bottom) with ExceptT (above), which means IO exceptions bypass the ExceptT handler.

9. Moderate: checkHeadingHierarchy Parses Tag Names Incorrectly
haskell, T.length name == 2
, T.head name == 'h'
, Just h <- [readMaybe (T.unpack $ T.drop 1 name) :: Maybe Int]
T.head is a partial function that throws on an empty Text. TagSoup can return empty tag names. This will crash on malformed HTML. Furthermore, T.length name == 2 filters headings up to h9 only — but it excludes nothing wrong here. The bigger issue is T.head on potentially empty text.

10. Moderate: Regex ReDoS Heuristics are Toy-Grade
haskellhasAlternationInRepetition :: String -> Bool
hasAlternationInRepetition s = "|" `L.isInfixOf` s && ")+" `L.isInfixOf` s
This is substring matching, not parsing. (a|b)c+ would falsely trigger (has | and )+ literally does not exist here, but ")+") can appear in unrelated positions). allSplits :: [a] -> [([a], [a])] generates O(n) splits and is called on every regex. The real solution is to use a library like safe-regex or call out to a ReDoS analyser. The claim of ReDoS protection here is marketing, not engineering.

12. Moderate: broadcastPatches Sends Full HTML Files Over WebSocket
haskellcontent <- TIO.readFile f
broadcastToPath clients (T.pack relPath) $ Aeson.object
  [ "type" .= ("patch" :: T.Text)
  , "html" .= content
  ]
This sends the entire HTML file content to the browser on every file change. For a page with embedded images rendered as base64 (LQIP), this could easily be hundreds of kilobytes per message, per client. Actual live reload systems send a reload signal and let the browser re-fetch. Or they do diffing. This "patch" mechanism sends a blob and presumably the client would replace the entire DOM — which would lose form state, scroll position, and any JS state. The client-side WebSocket handler in the dashboard HTML does not handle patch messages at all — only quality-seal.

SARA Audit — Part 2: Deep Continuation

17. Critical: remapMetadata Has a Two-Phase Race Condition — Remapping Is Always a No-Op
This is an architectural logic bug that goes unnoticed because match already returns an empty list (Issue #7), so nothing downstream ever exercises the remap path at scale. But the defect is independent and severe.
In SARA.hs:
haskell-- Phase 1: DSL execution — envRemapRules is []
let initialEnv = SaraEnv { ..., envRemapRules = [], ... }
result <- runExceptT $ runWriterT $ runReaderT (unSaraM m) initialEnv

-- Phase 2: Remap rules extracted AFTER DSL has already run
let allRemapRules = concat [ rs | RuleRemap rs <- rules ]
let finalEnv = initialEnv { envRemapRules = allRemapRules }
In DSL.hs, readMarkdown calls Remap.remapMetadata (envRemapRules env) — but it reads envRemapRules from the initialEnv, which always has envRemapRules = []. The RuleRemap collector runs in Phase 2, which is too late. The DSL is sequential — remapMetadata [...] tells the WriterT to emit a RuleRemap declaration, but readMarkdown has already consumed envRemapRules during Phase 1 of the same DSL execution.
In other words: the scaffolded site.hs template that ships with SARA has a metadata remapping step that silently does nothing every single time. The architecture document claims this is a first-class feature. It is broken by design.
The fix requires either a two-pass execution (first collect all RuleRemap, then re-run content rules with the populated env), or moving remap rules into a pre-execution configuration step entirely separate from the SaraM writer.

18. Critical: readMarkdown Reads Files Outside Shake's Dependency Graph
haskellreadMarkdownWith customHandler file = do
  content <- liftIO $ T.decodeUtf8 <$> BS.readFile file
This is a liftIO raw file read inside SaraM, which executes during the DSL evaluation phase — before Shake's build rules are even registered. It is completely invisible to Shake's dependency tracking. Shake has no idea this file was read, so:

Incremental rebuilds will not trigger when the file changes
There is no need [file] call, so Shake cannot parallelize correctly
The BLAKE3 hash oracle is registered in genRender's needBlake3, but by then the content has already been read in Phase 1 and cached in the Item — the hash check is checking the file at the Shake rule execution time against an Item built at DSL evaluation time, which could be a different version of the file if it changed between the two phases

This completely undermines the claimed incremental build feature. Hakyll avoids this problem by having readMarkdown be an Action (inside Shake's rule system), not a pre-rule DSL step.

20. Serious: Template Compilation Happens Outside the Oracle — No Caching
haskelltype instance RuleResult TemplateOracle = FilePath

addTemplateOracle = void $ addOracle $ \(TemplateOracle path) -> do
  -- ... audits, then returns the path
  pure path

renderTemplate tplPath ctx = do
  _ <- askOracle (TemplateOracle tplPath)   -- oracle: just checks audit
  res <- liftIO $ Mustache.compileMustacheFile tplPath  -- ← compilation happens HERE
  let result = TL.toStrict $ Mustache.renderMustache res ctx
  pure $ Right result
The oracle result is FilePath — it stores nothing. The template is compiled fresh on every single call to renderTemplate. For a 500-page site with one template, this is 500 compileMustacheFile calls, each doing a full parse of the template file, completely defeating Shake's oracle caching mechanism. The oracle exists only to run the security audit, not to cache the compiled template. The comment -- stache has its own cache if we use the same tpl object is present in the code but not acted on. This is either a deliberate deferral or a fundamental misunderstanding of how Shake oracles work.

21. Serious: Duplicate Error Type Hierarchy — SARA.Error and SARA.Routing.Error Define the Same Constructors
haskell-- SARA.Error:
RouteConflict :: { rtFile1, rtFile2, rtOutput } -> SaraError 'EKRouting
RouteRegexInvalid :: { rtPattern, rtDetail }    -> SaraError 'EKRouting

-- SARA.Routing.Error:
data RoutingError where
  RouteConflict   :: { rtFile1, rtFile2, rtOutput } -> RoutingError
  RouteRegexInvalid :: { rtPattern, rtDetail }      -> RoutingError
Two completely separate types with identical constructor names and fields exist in parallel. The routing engine returns RE.RoutingError, which the DSL then manually transcribes into SaraError 'EKRouting:
haskellLeft (RE.RouteRegexInvalid p d) -> Left $ RouteRegexInvalid p d
Left (RE.RouteConflict f1 f2 o) -> Left $ RouteConflict f1 f2 o
This is pure boilerplate that exists because the module structure was designed backwards. SARA.Routing.Error should not exist — resolveRoute should return Either (SaraError 'EKRouting) directly. The transcription layer is a maintenance hazard: any new field added to RoutingError must be mirrored in both places.

22. Serious: filterEvents Is Called Twice in watchProject
haskell_ <- watchTree mgr root (filterEvents) $ \event -> do
  when (filterEvents event) $ do   -- called again here

watchTree already takes a predicate as its third argument — it only calls the callback when the predicate returns True. 23. Serious: prop_yaml_roundtrip Property Test Will Always Fail on Non-Empty Input
haskellprop_yaml_roundtrip pairs =
  ...
  in case parseFrontmatter "test.md" content of
       Right (meta, _) -> Aeson.toJSON meta === Aeson.toJSON obj
       Left _          -> property False   -- ← fails on parse error
parseFrontmatter calls applyHtmlEscape which calls escapeHtml. But escapeHtml has the wrong operator ordering (Issue #5). For any alphanumeric-only input (which the test uses), there are no HTML special chars, so the roundtrip accidentally passes. 24. Serious: checkHeadingHierarchy Reports False Positives for Every Page Starting With h1
haskellin go headings 0   -- prev starts at 0
...
go (h:hs) prev =
  if h > prev + 1
  then ... SEOHeadingSkip ...
Starting with prev = 0, the first heading h1 satisfies 1 > 0 + 1 → False. Fine. But h2 as the first heading would be 2 > 1 → False. And h3 as first heading is 3 > 1 → True. So any page that starts with h3 or deeper will generate a false-positive "Heading Skipped from 0 to 3" warning, since 0 is not a real heading level at all. The initial prev should be 1 (the valid starting level), and the check should be h > prev + 1 && prev /= 0 or use Maybe Int for the initial state.

31. Moderate: checkInternalLinks Uses HS.member on Full Paths, But Site Graph Has Inconsistent Path Formats
