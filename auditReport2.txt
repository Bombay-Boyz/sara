Half-Fixed — New Problems Introduced 🔶
1. loadConfig Will Always Fall Back to Defaults — Field Name Mismatch
The FromJSON instance is derived generically:
haskelldata SaraConfig = SaraConfig
  { cfgSiteTitle       :: !Text
  , cfgSiteUrl         :: !Text
  ...
instance FromJSON SaraConfig  -- generic, no options
Aeson's generic FromJSON uses the literal record field names as JSON/YAML keys. So it expects a YAML file with keys cfgSiteTitle, cfgSiteUrl, cfgSiteAuthor, etc. But the scaffolded sara.yaml contains title, author, baseUrl, outputDir, defaultTemplate. These will never match. The decode will fail silently (Left _ → return defaultConfig) and every user site will use hardcoded defaults. The fix is either fieldLabelModifier with stripPrefix "cfg" + camelToKebab, or a manual parseJSON instance, or rename the record fields.
2. validateSEO Now Throws on Missing description — Breaking Change With No Migration Path
haskell(_, Nothing) -> throwError [AnySaraError $ SEODescriptionMissing (itemPath item)]
The previous validateSEO was a no-op (a bug). The fix is real — it now validates. But requiring description in frontmatter means every existing user's posts will fail the build immediately with no graceful degradation path. Hugo and Hakyll treat missing description as a warning, not a build error. This is now a hard failure during the planning phase (envIsPlanning = True), which means even the skeleton items created during planning will throw if description is absent. The planning-phase items have itemMeta = KM.empty, so every file always fails validation during planning. The envIsPlanning guard on validateSEO handles title and description checks only in the non-planning branch, but this means planning always succeeds regardless — the validation is deferred to execution. The logic is now complex and the interaction between planning/execution phases and validation is difficult to reason about.
3. MonadWriter Instance Has error for listen and pass
haskellinstance MonadWriter [RuleDecl] SaraM where
  tell ds = ...
  listen = error "listen not implemented for SaraM"
  pass   = error "pass not implemented for SaraM"
This is a partial typeclass instance — a new class of unsafety introduced in this revision. listen and pass are in the MonadWriter typeclass. Any library function that calls listen or pass on a MonadWriter constraint will cause a runtime crash with no compile-time warning. The correct approach is to not implement MonadWriter at all and instead define tellRule :: RuleDecl -> SaraM () as a plain function, removing the typeclass dependency. Or define a newtype class that doesn't pretend to be full MonadWriter.
4. templateCache is unsafePerformIO — Still Present, But Now Justified
haskell{-# NOINLINE templateCache #-}
templateCache :: IORef (Map.Map FilePath Mustache.Template)
templateCache = unsafePerformIO $ newIORef Map.empty
This is the legitimate use of unsafePerformIO (top-level IORef with NOINLINE). The comment acknowledges the justification. However, the cache has a race condition: readIORef and the subsequent atomicModifyIORef' are not atomic as a pair. Two Shake threads rendering the same template simultaneously will both find a cache miss and both compile the template — the second write wins, which is harmless for correctness, but wastes work. Use STM with TVar and modifyTVar' under atomically, or use a MVar with double-checked locking. For a pure cache with no side effects from double compilation, the current behavior is acceptable in practice, but it is worth noting.
5. route DSL Function Still Has a TODO_RESOLVE Placeholder
haskellroute r item = return $ item { itemRoute = ResolvedRoute "TODO_RESOLVE" } -- Placeholder
This is a shipped placeholder in a public API function. Any user calling route SlugRoute item will get an item with the literal path "TODO_RESOLVE" as its output path, silently producing a file named TODO_RESOLVE in _site. The actual routing logic (resolveRoute) exists and works — it just isn't connected here. This should call liftIO $ resolveRoute r (itemPath item) and handle the result.
6. genDataDependency Is Still a No-Op — loadData Shake Tracking Still Broken
haskellgenDataDependency :: SaraEnv -> FilePath -> Rules ()
genDataDependency _ path = do
  -- ... long comment explaining why it's still not implemented ...
  return ()
loadData tells Shake about a RuleDataDependency, but the planner converts that to nothing. The DataOracle exists in Hash.hs and is correct, but it's never connected to the actual rebuild trigger. A data file changing will not cause a rebuild. The comment in the code admits this and defers to "rely on the oracle for now" — but nothing calls askData inside any build rule, so the oracle never fires for data files.
7. tell in MonadWriter Stores Rules in Reverse — Reversal Applied Twice
haskell-- Monad.hs:
tell ds = atomicModifyIORef' ref (\rules -> (reverse ds ++ rules, ()))

-- SARA.hs:
rules <- reverse <$> readIORef rulesRef
tell prepends reverse ds (reversing a singleton list is a no-op, but reversing multi-item ds is not). Then readIORef reverses again. The intent is to produce rules in declaration order. But reverse ds ++ rules builds a left-fold list — appending to the front means the final reverse gives declaration order only if ds is always a singleton. When tell is called with a multi-item list (which the current code never does, but could), the order would be wrong. This is fragile. Use Seq or DList, or simply ds ++ rules with a final reverse, not reverse ds ++ rules.

Remaining Unaddressed Issues 🔴
Issue #3 — validateSEO During Planning Creates Empty Items
The planning-phase path:
haskellif envIsPlanning env
  then return $ Item { ..., itemMeta = KM.empty, itemBody = "", ... }
An item with itemMeta = KM.empty and itemBody = "" is what gets passed to render during planning. These empty items are then stored as RuleRender tpl emptyItem outPath declarations. When Shake executes genRender, it calls askItem env (itemPath item) to reload the real item — that's the new Oracle. This is an improvement. But the empty planning-phase items are also passed to buildSearchIndex, buildSitemap, and buildRSS — those get lists of empty items with no metadata and empty bodies. During Shake execution, genSearch/genSitemap/genRSS iterate over these stored items and call mkSearchEntry, mkUrl, etc., on empty metadata. The sitemap will have blank URLs; the search index will have empty documents; the RSS feed will have untitled items with empty descriptions. The askItem pattern is only wired up for genRender and genPartialSearch — not for sitemap or RSS.
Issue #12 — Live reload still sends full HTML
broadcastPatches is unchanged — still sends full HTML file contents per client per changed file.
Issue #31 — Link checker path format
genDiscover now inserts outPath (relative, without outDir prefix) into the site graph:
haskellliftIO $ atomicModifyIORef' (envSiteGraph env) $ \sg -> (HS.insert src sg, ())
But genRender inserts outPath (also relative). The link checker normalizes "/posts/hello.html" to "posts/hello.html" and checks membership. This is closer to correct, but the consistency depends on whether all insertion sites use the same format. A full audit of every HS.insert call would be needed to confirm.

New Issues Not Present Before
N1. match Runs DSL Functions During Planning With Real File I/O
haskellmatch g f = do
  let patStr = T.unpack (unGlobPattern g)
  files <- liftIO $ globDir1 (compile patStr) "."
  items <- mapM f files    -- ← runs f for every file during planning
  tell [RuleMatch g f]
  return items
mapM f files runs f (the user's compiler function) during the planning phase. f calls readMarkdown which checks envIsPlanning and returns an empty item. This means f is called twice for every file: once during planning (returning empty items) and once during Shake execution (via ItemOracle). Any side effects in f beyond reading files — e.g., liftIO calls, logging — will fire twice per file. For a 1000-page site, this is 2000 DSL function executions.
N2. SaraM's WriterT Layer Was Removed But MonadWriter Is Still Implemented
The monad stack changed from ReaderT ... (WriterT [RuleDecl] (ExceptT ...)) to ReaderT ... (ExceptT ... IO), which is correct. But then MonadWriter was manually reimplemented via IORef. This means:

WriterT semantics (atomic tell+result) are gone
But the MonadWriter typeclass is still in scope, pulling in all its laws and friends
Code that imports Control.Monad.Writer (tell) now calls the IORef variant, which is not the same as WriterT tell — specifically, if throwError is called after tell, in the old stack the tells would be discarded (ExceptT semantics); in the new stack the IORef mutations persist even if the computation fails. This is a semantic change that could lead to phantom rules being registered from failed sub-computations.
