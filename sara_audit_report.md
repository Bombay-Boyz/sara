# SARA Static Site Generator — Critical Audit Report

> **Mission:** Be the world's most adopted static site generator — offering Hakyll's type safety with Hugo's (Rust-level) runtime performance.
>
> **Reviewer:** Independent code audit, April 2026
> **Repository:** https://github.com/Bombay-Boyz/sara
> **Commit audited:** HEAD (main branch)

---

## Executive Summary

SARA is architecturally ambitious and has several genuinely impressive subsystems — a well-designed GADT error type, a clean phantom-type pipeline for validation states, Blake3 content hashing, and a thoughtful security guard layer. The documentation is exemplary.

However, the gap between what the project *documents* and what the code *actually does* is severe. Several features described as core capabilities are either no-ops, stubs, or missing entirely. The most critical finding is that **`genMatch` — the Shake-level translation of the DSL's primary combinator — is a complete no-op** (`return ()`). This alone means SARA cannot rebuild content files incrementally; it is currently a planning-phase simulator, not a working build engine.

Additionally, multiple partial functions, unsafe I/O primitives, race conditions, and phantom-type violations undermine the "100% type safety and compilation guarantee" claim. Compared to Hugo and Hakyll, significant features are absent.

The project needs a correctness-first sprint before performance or adoption work proceeds.

| **L-01** | **`genMatch` is a no-op** (`SARA/Internal/Planner.hs:genMatch _ _ _ = return ()`). The DSL's most important combinator — `match` — collects items at planning time but its Shake-level translation does absolutely nothing. No Shake rules are ever registered for content files. | Content rebuilds are never tracked by Shake's dependency graph. Incremental rebuilds are impossible. The entire "fast like Hugo" claim is void because Shake cannot schedule parallel work it doesn't know about. Users who change a post will trigger a full re-run, not a targeted rebuild. | `genMatch` must register a Shake rule: for each file matching the glob, register `outPath %> \o -> do { need [src]; item <- askItem env src; writeFile' o (renderItem item) }`. The planning pass already populates `stateItemCache` — use it. | **Existential.** This is the core build loop. Nothing else matters until this is fixed. |
| **L-02** | **`site.hs` is never compiled or executed** (`app/Main.hs:runDefaultBuild`). Despite comments saying "Using project site.hs…", the code always runs `defaultSite` — a hardcoded pipeline. The entire user-defined DSL entrypoint is dead. | SARA's primary differentiator over Hugo (user-written Haskell build logic) does not work. Any `site.hs` a user writes is silently ignored. This is the single biggest gap vs Hakyll. | Use `System.Process` or a GHCi interpreter (e.g., `hint` library) to compile and execute `site.hs` at runtime, similar to how Hakyll runs `site build`. | **L-03** | **`validateSEO` coerces type without validation in planning mode** (`SARA/DSL.hs:validateSEO`). During the planning pass (`envIsPlanning = True`), the function returns an `Item 'Validated` by field-copying without running any SEO checks. The phantom type `'Validated` is obtained for free. | The type system guarantee — that only items passing SEO validation can be rendered — is broken. Malformed content with no title or description can reach `render` undetected. In production builds this is caught, but any code path that uses planning-mode items as though they are fully validated is silently unsafe. | Separate the type-coercion path from the validation path. During planning, either (a) always treat items as `'Unvalidated` and only promote after the execution pass runs real checks, or (b) introduce a third phantom state `'Planning` to make the distinction explicit at the type level. | **High.** Undermines the core type-safety value proposition. |
| **L-04** | **`guardPath` uses text prefix matching** (`SARA/Security/PathGuard.hs`): `T.isPrefixOf (T.pack normRoot) (T.pack normCandidate)`. | A project root at `/home/user/site` would incorrectly *pass* the guard for `/home/user/site2/../../etc/passwd` if `site2` normalises to a path sharing the prefix, depending on OS normalization. The correct approach uses canonical path comparison, not string prefix matching. Also, the guard does not call `canonicalizePath` on the candidate — only on the root during `mkProjectRoot`. A symlink in the project can bypass this. | Use `canonicalizePath` on both root and candidate inside `guardPath` (requires `IO`, so adjust the signature), then compare with `makeRelative` and check that the result does not start with `..`. | **High.** Security guarantee is advertised but partially hollow. |
| **L-05** | **`loadConfig` silently swallows YAML parse errors** (`SARA/Config.hs`): `Left _ -> return defaultConfig`. | A malformed `sara.yaml` causes SARA to silently build with default config (wrong title, wrong base URL, wrong output dir). Users will deploy a broken site with no error. | Pattern match on the `Left` case, print a human-readable parse error (using the existing `SaraError 'EKConfig` machinery), and `exitFailure`. | **High.** Violates the "Human-First Error Messages" promise. |
| **L-06** | **`remapMetadata` is total-failure-on-any-missing-key** (`SARA/Frontmatter/Remap.hs`). `foldr remap (Right obj) rules` — if any `from` key is absent, the whole remap fails with an error. | Migration from Hugo/Jekyll sites where metadata varies per file (some posts have `published_at`, others don't) will fail entirely. Every post must have every remapped key, which is incompatible with the "Migration-First" philosophy. | Make missing `from` keys a no-op (return the object unchanged) or a warning, not an error. Add an `optional` flag to the remap config. | **High.** Directly breaks the migration use case. |
| **L-07** | **`checkHeadingStructure` uses `read`** (`SARA/SEO/Audit.hs`): `read (T.unpack (T.drop 1 (getTagName t))) :: Int`. `read` is a partial function that throws an exception on non-numeric input. | Any HTML tag whose name starts with `h` and has length 2 but whose second character is not a digit (e.g., `hr`, rendered via a shortcode or raw HTML) will crash the SEO audit with an unhandled exception, halting the build with no useful error message. | Replace `read` with `readMaybe` and filter out `Nothing` results, or restrict the regex for heading detection to `/^h[1-6]$/` before numeric parsing. | **High.** Build crashes on valid HTML inputs. |
| **L-08** | **`SEOTitleMissing` is a hard error but `SEODescriptionMissing` is a warning** (`SARA/DSL.hs:validateSEO`). This asymmetry is undocumented. | Users get confused why missing `title` halts the build but missing `description` only prints a warning. The inconsistency is not explained in any error code documentation. | Make both configurable via an `AuditLevel` setting in `sara.yaml`. Document the defaults. | **Medium.** UX inconsistency. |
| **L-09** | **`injectLQIPs`/`findLQIPTokens` fragile token parsing** (`SARA/Internal/Planner.hs`). Token format `__LQIP__:path__` is parsed with `T.drop 2 after` which assumes exactly two underscores follow the path. | Any image path containing `__` (e.g., `images/hero__2x.jpg`) will corrupt the token, silently producing malformed HTML with a broken background-image URL. | Use a unique delimiter that cannot appear in file paths, or use a proper parser (e.g., megaparsec) with start/end markers. Consider a UUID-based token per image instead of path-derived tokens. | **Medium.** Silent data corruption in output HTML. |
| **L-10** | **Race condition in `askItem`** (`SARA/Internal/Planner.hs`). The ItemOracle writes to `stateItemCache` via `atomicModifyIORef'`, then `askItem` reads from `stateItemCache` in a separate `readIORef` call outside the Oracle's critical section. | Under Shake parallelism, two threads could both miss the cache, both run the Oracle, and produce inconsistent results. More critically, the cache-write and the subsequent cache-read by the calling thread are not atomic, so a third thread writing between those two operations could cause a stale read. | Either (a) use an `MVar` per item (as is done for templates) for proper locking, or (b) return the compiled `Item` directly from the Oracle result type (`type instance RuleResult ItemOracle = ...`) instead of using a side-channel IORef cache. Option (b) is cleaner and idiomatic Shake. | **High.** Non-deterministic build results under parallelism. |
| **U-01** | **`T.decodeUtf8` is a partial function** (`SARA/Monad.hs:readTextFileTracked`): `T.decodeUtf8 <$> BS.readFile`. | Any source file containing invalid UTF-8 bytes (common in migrated legacy sites, files with BOMs, or files edited on Windows) will throw an unhandled `DecodeError` exception that bypasses SARA's error handling entirely, crashing with a GHC runtime error. | Replace with `T.decodeUtf8'` and handle the `Left` case, emitting a `SaraError 'EKFrontmatter` with the file path and byte offset. | **Critical.** Crash on valid user input with no recovery path. |
| **U-02** | **`unsafePerformIO` for template cache** (`SARA/Template/Renderer.hs`): `let m = unsafePerformIO (newMVar Nothing)`. | `unsafePerformIO` breaks referential transparency and is only safe under strict conditions (memoisation of pure values). Here it is used to lazily create an `MVar` inside a `atomicModifyIORef'` block. GHC may float this expression, duplicate it, or evaluate it at unexpected times, leading to multiple MVars being created for the same template, race conditions, and incorrect caching. | Remove `unsafePerformIO`. Pre-populate the template cache MVar map at the start of `runBuild` by collecting all template paths from `RuleRender` declarations. This is safe, pure, and works correctly under parallelism. | **Critical.** Undefined behaviour in concurrent builds. |
| **U-03** | **`unsafePerformIO` for global client counter** (`SARA/LiveReload/Server.hs`): `clientCounter = unsafePerformIO $ newIORef 0`. | The `{-# NOINLINE #-}` pragma mitigates the worst risks here, but this is still an anti-pattern. Under GHC 9.x with `-O2`, the compiler may still behave unexpectedly. The counter is shared global mutable state without explicit module-level initialisation. | Pass the `IORef Int` as a parameter through `startLiveReloadServer` and thread it through `wsApp`, eliminating the global. | **Medium.** Code smell; potential correctness issue with optimisation flags. |
| **U-04** | **`hasAlternationInRepetition` logic is inverted** (`SARA/Security/RegexGuard.hs`). The branch `| c == '|' = (depth > 0) || go cs depth True` short-circuits on `depth > 0`, returning `True` for any pipe character found inside a group regardless of whether there is a subsequent quantifier. | The ReDoS heuristic produces false positives (legitimate regexes blocked) and false negatives (genuinely dangerous patterns allowed). For example, `(a|b)+` is dangerous but the logic may not catch it correctly; `(a|b)c` is safe but may be rejected. | Use an established ReDoS detection library or a proper regex complexity analyser. At minimum, rewrite this as a state machine with correct depth tracking that only fires when the closing `)` is followed by `+` or `*`. Add property tests covering known-dangerous patterns. | **High.** Security guarantee is false. |
| **U-05** | **`FrontmatterParseFailure` always reports `SourcePos path 1 1`** (`SARA/Frontmatter/Parser.hs`). YAML/TOML parse errors carry line/column information in the underlying library's error type, but it is discarded. | Developers cannot find the broken line in their frontmatter. Error messages always say "line 1, column 1" regardless of where the actual parse failure occurred. This directly contradicts the "pinpoint the exact line" promise. | Extract `Data.Yaml.ParseException` fields (which include `problemMark`) and `Toml.ParseError` (which includes offset information) to populate `SourcePos` accurately. | **Medium.** Breaks "Human-First Error Messages" promise. |
| **C-01** | **No parallelism exposed to users or configurable** (`SARA/Internal/Engine.hs`). Shake's `shakeOptions` does not set `shakeThreads`, which defaults to one thread. The `--jobs` flag is not exposed in the CLI. | All builds are single-threaded despite the "saturate all CPU cores" claim in the documentation. Image processing, template rendering, and search index generation run sequentially. | Set `shakeThreads = 0` in `shakeOptions` (Shake's convention for "use all cores"), and expose `--jobs N` in the CLI's `build` subcommand. | **Critical.** The "Rust-level performance" claim is impossible with single-threaded execution. |
| **C-02** | **`match` runs sequentially in the planning phase** (`SARA/DSL.hs`): `items <- mapM f files`. The per-file compiler function is run with `mapM` (sequential) not `mapConcurrently`. | Even if `shakeThreads` is fixed, the DSL planning phase itself processes all matched files one-by-one. For a 5,000-post site this is a significant bottleneck. | Replace `mapM f files` with `mapConcurrently f files` from `UnliftIO.Async`, or defer all per-file work to the Shake execution phase (which is parallel by design) and only collect paths in the planning phase. | **High.** Contradicts the "10k Post Test" benchmark target. |
| **C-03** | **`mergePartialIndexes` reads all partial indexes sequentially** (`SARA/Search/Index.hs`): `mapM (liftIO . Aeson.decodeFileStrict') partialPaths`. | On large sites with thousands of pages, the search index merge step is a serial I/O bottleneck that cannot be parallelised as written. | Use `mapConcurrently` for the read phase, then aggregate the results. Alternatively, use a streaming fold. | **Medium.** Performance bottleneck at scale. |
| **C-04** | **`watchProject` uses `threadDelay 1000000` to keep alive** (`SARA/LiveReload/Watcher.hs`). The main watch thread blocks on `forever $ threadDelay 1000000` (1 second sleeps) instead of using a proper concurrency primitive. | On shutdown (SIGINT), the watcher takes up to 1 second to respond. More importantly, `withManager` + `forever` means the watcher never propagates exceptions — if the build crashes, the watcher stays alive silently. | Replace the spin loop with `takeMVar` on a shutdown MVar, and use `async`/`withAsync` for structured concurrency. | **Medium.** Poor shutdown behaviour; masks build errors in dev mode. |
| **T-01** | **`{-# OPTIONS_GHC -Wno-partial-fields #-}`** is present in `SARA/Types.hs` and `SARA/Error.hs`. These pragmas suppress GHC warnings about GADT record selectors that are only valid for some constructors. | Calling `lrPath` on a `SlugRoute` constructor, or `rtPattern` on a `RouteConflict`, will throw a runtime exception. GHC would normally warn about this. The pragma silences those warnings, removing the compile-time safety net. | Remove record selectors from GADT constructors entirely, or use pattern matching exclusively. If record syntax is needed, use `RecordWildCards` only on non-GADT types. | **High.** Removes GHC's protection against partial field access. |
| **T-02** | **No `-Werror` in CI configuration** despite being mandated as "non-negotiable" in `SARA_Architecture_and_Implementation_Guide_v1.1.md §21`. New contributors add code with `head`, `fromJust`, partial patterns, etc., and these ship to users. ## Priority Order for Remediation

Given the mission ("world's most adopted SSG"), the following sequence is recommended:

**Sprint 1 — Make It Work (Correctness)**

Fix L-01 (`genMatch`), L-02 (`site.hs` execution), L-03 (`validateSEO` coercion), U-01 (`T.decodeUtf8`), U-02 (`unsafePerformIO` in template renderer), and C-01 (single-threaded Shake). Nothing else can be evaluated until the build engine actually builds.

**Sprint 2 — Make It Safe**
